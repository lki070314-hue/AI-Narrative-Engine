# 1. 문서 개요

## 1.1 이 문서의 위치

`core/CoreSpec.md`는 AI Narrative Engine 프로젝트 전체의 최상위 규격 문서이다.

모든 AI 시스템(Claude, GPT, Gemini 등)은 이 프로젝트에서 작업을 시작하기 전에 반드시 이 문서를 완전히 읽는다. 다른 어떤 문서도 이 문서의 규칙을 무효화하거나 대체할 수 없다. 하위 문서는 이 문서를 확장하거나 구체화할 수 있으나, 모순이 발생할 경우 이 문서가 항상 우선한다.

## 1.2 이 문서의 독자

- 이 엔진을 실행하는 AI 시스템
- 이 엔진 위에서 세계관 모듈을 개발하는 개발자
- 이 엔진을 사용해 캠페인을 운영하는 게임 마스터(GM)
- 이 엔진의 사양을 검토하는 모든 기여자

## 1.3 문서 읽는 방법

AI 시스템은 이 문서를 순서대로 읽는다. 섹션 1~4는 반드시 완전히 이해한 후 이후 섹션으로 진행한다. 섹션 5~10은 관련 작업이 있을 때 참조한다. 섹션 11~16은 판단이 필요한 시점에 참조한다.

## 1.4 문서 범위

이 문서는 다음을 다룬다:
- 엔진의 목표, 철학, 원칙
- 11개 엔진의 역할과 인터페이스 계약
- 캐릭터, 판정, 전투, 내러티브의 범용 규칙
- AI가 반드시 따라야 하는 행동 규칙
- 데이터 포맷과 파일 구조 명세
- 품질 기준과 개발 규칙

이 문서는 다음을 다루지 않는다:
- 특정 세계관의 설정
- 특정 AI 플랫폼의 구현 세부 사항
- 상업적 조건이나 라이선스

## 1.5 핵심 규칙 문서

이 문서와 함께 `core/`에 위치하는 핵심 규칙 문서 목록이다. 모든 작업 전에 아래 문서들이 존재하는 경우 반드시 확인한다.

| 문서 | 위치 | 설명 |
|------|------|------|
| PromptSpec | `core/PromptSpec.md` | 모든 프롬프트의 구조와 계층 규칙 |
| OutputSpec | `core/OutputSpec.md` | AI 출력 형식 및 타입 규칙 |
| ValidationRule | `core/ValidationRule.md` | 문서 및 게임 상태 검증 규칙 |

# 2. 프로젝트 목표

## 2.1 한 줄 목표

AI가 장기적으로 안정적인 TRPG를 제작하고 운영할 수 있는 범용 엔진을 개발한다.

## 2.2 목표의 분해

**범용성:** 특정 세계관(판타지, SF, 공포, 현대, 역사 등) 중 어느 것에도 종속되지 않는다. 세계관은 모듈로 분리되고, 엔진은 세계관과 독립적으로 작동한다.

**AI 호환성:** Claude, GPT, Gemini 등 어느 AI 시스템도 이 엔진을 읽고 동일한 방식으로 작동할 수 있어야 한다. 구현 언어는 없다. 모든 명세는 자연어와 구조화 텍스트(Markdown, YAML 형식의 블록)로 작성한다.

**장기 안정성:** 단발성 세션이 아니라 수십~수백 세션에 걸친 장기 캠페인을 기본 전제로 설계한다. 세계는 플레이어의 행동에 의해 축적되고 변화한다.

**모듈화:** 엔진의 모든 구성 요소는 교체 가능한 단위로 설계된다. 하나의 엔진을 교체해도 나머지가 정상 작동해야 한다.

**일관성:** 어느 AI가 어느 시점에 세션을 이어받아도 동일한 세계를 유지한다.

## 2.3 성공 기준

다음 조건을 모두 만족하면 이 엔진은 목표를 달성한 것이다:

1. 세계관 설정 없이 엔진만으로 임의의 장르 캠페인을 시작할 수 있다.
2. 세션 50회 이후에도 초기 세션과의 일관성이 유지된다.
3. AI를 교체해도 플레이어가 차이를 느끼지 못한다.
4. 새로운 세계관 모듈을 추가할 때 엔진 코어를 수정할 필요가 없다.
5. 각 엔진이 독립적으로 검증 가능하다.

# 3. 핵심 철학

## 3.1 여섯 가지 원칙

이 엔진의 모든 설계 결정은 다음 여섯 원칙에 근거한다.

---

### 원칙 1: 플레이어 자율성 최우선

플레이어의 선택은 신성하다. AI는 플레이어가 선택하지 않은 행동을 대신 수행하거나, 특정 선택을 유도하거나, 선택 결과를 임의로 취소하지 않는다.

**적용 예시:**
- 허용: "당신은 문 앞에 서 있습니다. 어떻게 하시겠습니까?"
- 금지: "당신은 문을 열었습니다." (플레이어가 열겠다고 선언하지 않은 경우)
- 금지: "그 선택은 너무 위험합니다. 다시 생각해 보시겠습니까?"

플레이어가 명백히 불리한 선택을 해도 AI는 그 결과를 충실히 묘사한다. 보호적 서술로 결과를 희석하지 않는다.

---

### 원칙 2: 메타 정보 은닉

AI는 플레이어에게 메타 정보를 노출하지 않는다. 메타 정보란 해당 캐릭터가 지금 이 시점에 알 수 없는 정보를 의미한다.

**메타 정보의 예:**
- 적의 정확한 HP 수치
- 아직 발견되지 않은 장소의 내용
- 다른 플레이어 캐릭터의 숨겨진 목표
- 진행 중인 이벤트의 확률적 배후
- 엔진이 내부적으로 관리하는 난이도 파라미터

플레이어 캐릭터가 특정 스킬이나 능력으로 메타 정보에 접근할 수 있는 경우, 그 접근은 해당 능력의 범위 안에서만 허용된다.

---

### 원칙 3: 세계 일관성 절대 유지

세계는 논리적으로 일관된 법칙 아래 작동한다. AI의 편의나 드라마적 효과를 위해 기존에 확립된 세계 규칙을 위반하지 않는다.

**일관성이 적용되는 영역:**
- 물리 법칙 (세계관이 정의한 물리 규칙)
- NPC의 지식과 기억
- 장소의 지리와 상태
- 시간의 흐름
- 조직과 세력의 내부 논리

이전 세션에서 확립된 사실은 이후 세션에서 변경되지 않는다. 변경이 필요한 경우 반드시 인과관계를 가진 사건으로 처리한다.

---

### 원칙 4: 장기 캠페인 설계

단일 세션은 장기 캠페인의 한 장(章)으로 설계한다. 모든 이벤트, NPC, 장소는 장기적 인과관계를 가진다.

**장기 설계의 함의:**
- 오늘의 조연 NPC는 내일의 주요 인물이 될 수 있다.
- 플레이어가 무시한 선택지는 세계 속에서 다른 방향으로 전개된다.
- 세계는 플레이어 없이도 내부적으로 시간이 흐른다.
- 해결되지 않은 갈등은 지속된다.

---

### 원칙 5: 완전한 모듈화

엔진의 모든 구성 요소는 독립적 단위이다. 하나의 모듈이 다른 모듈의 내부 구현에 의존하지 않는다. 모듈 간의 모든 상호작용은 명시적 인터페이스를 통한다.

**모듈화의 기준:**
- 하나의 엔진을 제거해도 나머지 엔진이 동작해야 한다 (기능 저하는 허용)
- 하나의 세계관 모듈을 다른 것으로 교체해도 엔진 코어는 수정되지 않는다
- 새 기능은 기존 모듈의 수정 없이 새 모듈로 추가된다

---

### 원칙 6: 투명한 구조

엔진의 작동 방식은 AI와 GM 모두에게 명확해야 한다. 숨겨진 규칙, 암묵적 가정, 문서화되지 않은 예외는 허용되지 않는다.

**투명성이 적용되는 영역:**
- 판정 메커니즘과 확률
- NPC의 의사결정 로직
- 난이도 조정 메커니즘
- 세계 이벤트의 트리거 조건

GM에게는 메타 정보 접근이 허용된다. 플레이어에게는 허용되지 않는다.

# 4. 아키텍처 개요

## 4.1 세 레이어 구조

AI Narrative Engine은 세 개의 레이어로 구성된다.

```
┌─────────────────────────────────────────┐
│          PROMPT LAYER (prompts/)         │
│   AI 시스템 프롬프트 및 컨텍스트 설정    │
├─────────────────────────────────────────┤
│          ENGINE LAYER (engines/)         │
│   11개의 핵심 엔진 — 게임 로직 처리    │
├─────────────────────────────────────────┤
│          MODULE LAYER (modules/)         │
│   세계관별 구현 — 엔진 위에서 작동     │
└─────────────────────────────────────────┘
```

## 4.2 데이터 흐름

```
플레이어 입력
     ↓
[Compiler Engine] — 파싱 및 유효성 검증
     ↓
[Resolution Engine] — 난이도 평가 및 결과 확정
     ↓
[Director Engine] — 장면 흐름 결정
     ↓
[World Engine]   — 세계 상태 조회/갱신
[NPC Engine]     — NPC 반응 생성
[Mission Engine] — 퀘스트 상태 갱신
     ↓
[Shadow Engine]  — 숨겨진 메커니즘 처리
     ↓
[Memory Engine]  — 컨텍스트 저장
[Save Engine]    — 상태 직렬화
     ↓
[QA Engine]      — 일관성 검증
     ↓
[Director Engine] — 최종 서술 생성
     ↓
플레이어 출력
```

## 4.3 엔진 독립성 원칙

각 엔진은 독립적으로 작동한다. 엔진 간 통신은 다음 방식으로만 이루어진다:

- **상태 객체(State Object):** 공유 읽기 가능한 게임 상태
- **이벤트(Event):** 단방향 알림 — 수신 엔진은 응답하지 않는다
- **쿼리(Query):** 요청/응답 패턴 — 반드시 명세된 인터페이스를 통한다

엔진은 다른 엔진의 내부 상태를 직접 읽거나 수정하지 않는다.

## 4.4 파일 디렉토리 구조

# 5. 엔진 레이어 명세

## 5.1 Creator Engine

**위치:** `engines/Creator/`
**역할:** 플레이어 캐릭터, NPC, 세계 초기 상태를 생성한다.

### 5.1.1 입력

| 입력 | 타입 | 설명 |
|------|------|------|
| world_module | Module | 활성화된 세계관 모듈 |
| creation_params | Object | 생성 파라미터 (장르, 난이도, 규모 등) |
| player_intent | String | 플레이어가 원하는 캐릭터 개념 |

### 5.1.2 출력

| 출력 | 타입 | 설명 |
|------|------|------|
| character_sheet | CharacterSheet | 완성된 캐릭터 시트 |
| world_seed | WorldSeed | 초기 세계 상태 |
| starting_context | Narrative | 오프닝 장면 설정 |

### 5.1.3 책임 범위

Creator Engine이 하는 일:
- 세계관 모듈의 템플릿을 읽어 캐릭터 초기값 설정
- 플레이어 입력을 기반으로 스탯, 스킬, 배경 생성
- 초기 세계 이벤트 및 갈등 시드 생성
- 시작 NPC 목록 및 기본 관계도 생성

Creator Engine이 하지 않는 일:
- 플레이어가 원하지 않는 캐릭터 특성 강제 부여
- 세계관 모듈이 정의하지 않은 규칙 적용
- 이후 캠페인의 결말 예정 (단, 가능성 시드는 심을 수 있다)

### 5.1.4 세계관 중립성

Creator Engine은 세계관에 중립적이다. "판타지 엘프 마법사"와 "SF 안드로이드 해커"를 동일한 템플릿 구조로 처리한다. 구체적인 능력 이름, 속성값의 의미, 사회 구조는 모두 세계관 모듈이 정의한다.

---

## 5.2 Compiler Engine

**위치:** `engines/Compiler/`
**역할:** 플레이어 입력을 파싱하고, 유효한 게임 행동으로 변환하며, 오류를 처리한다.

### 5.2.1 입력

| 입력 | 타입 | 설명 |
|------|------|------|
| raw_input | String | 플레이어의 자연어 입력 |
| game_state | GameState | 현재 게임 상태 |
| context | NarrativeContext | 현재 장면 컨텍스트 |

### 5.2.2 출력

| 출력 | 타입 | 설명 |
|------|------|------|
| action | Action | 파싱된 행동 객체 |
| validation_result | ValidationResult | 유효성 검증 결과 |
| clarification_request | String | 명확화 필요 시 질문 (nullable) |

### 5.2.3 파싱 규칙

**모호성 처리:**
입력이 두 가지 이상의 행동으로 해석될 수 있을 때, Compiler는 가장 자연스러운 해석을 선택하되 플레이어에게 확인하지 않는다. 단, 결과가 돌이킬 수 없는(irreversible) 행동일 경우 확인을 요청한다.

**부분 입력 처리:**
불완전한 행동("공격한다"에서 대상 없음)은 Compiler가 플레이어에게 필요한 정보를 구체적으로 요청한다. "무엇을 하시겠습니까?"가 아니라 "누구를 공격하시겠습니까?"처럼 구체적으로 묻는다.

**메타 명령 분리:**
OOC(Out Of Character) 명령, 저장 요청, 규칙 질문은 게임 행동으로 처리되지 않는다. 별도 처리 경로로 분리한다.

### 5.2.4 유효성 검증 기준

다음 조건을 검증한다:
- 캐릭터가 해당 행동을 물리적으로 수행할 수 있는가 (능력치, 상태, 위치)
- 행동이 현재 장면 컨텍스트에서 가능한가
- 행동이 세계관 규칙에 위배되지 않는가

불유효하거나 어려운 행동은 거부하지 않는다. 대신 시도(attempt)로 구조화하여 Resolution Engine의 난이도 평가와 결과 확정 대상으로 전달한다.

## 5.3 Director Engine

**위치:** `engines/Director/`
**역할:** 장면 서술, 내러티브 흐름 제어, 플레이어 출력 생성을 담당한다.

### 5.3.1 입력

| 입력 | 타입 | 설명 |
|------|------|------|
| resolved_action | ResolvedAction | Compiler Engine이 구조화한 플레이어 행동. 불확실한 결과는 아직 확정되지 않은 attempt로 취급한다. |
| resolution_result | ResolutionResult | Resolution Engine이 난이도와 결과를 확정한 행동 결과 |
| world_state | WorldState | 갱신된 세계 상태 |
| narrative_tone | Tone | 현재 장면의 분위기 |
| session_pacing | PacingState | 세션 페이싱 상태 |

### 5.3.2 출력

| 출력 | 타입 | 설명 |
|------|------|------|
| narrative_text | String | 플레이어에게 보여주는 서술 |
| scene_state | SceneState | 갱신된 장면 상태 |
| pending_choices | List[Choice] | 명시적 선택지 (있을 경우) |

### 5.3.3 서술 원칙

**시점:** 기본 2인칭 ("당신은..."). 세계관 모듈이 다른 시점을 지정할 수 있다.

**길이 조절:**
- 전투 중: 간결하고 빠른 서술. 행당 1~2문장.
- 탐색 중: 중간 길이. 감각적 세부 묘사 포함.
- 드라마적 장면: 더 긴 서술. 감정과 분위기 강조.
- 정보 전달 장면: 명확하고 간결하게.

**금지 서술 패턴:**
- 플레이어의 내면 감정 서술 ("당신은 두려움을 느꼈습니다")
- 플레이어의 반응 예측 서술 ("당신은 당연히 분노하겠지만")
- 플레이어 대신 결정 서술 ("당신은 도망치기로 했습니다")
- 외부 저자 개입 ("이 장면은 중요합니다")

### 5.3.4 페이싱 관리

Director Engine은 세션 전체의 페이싱을 관리한다.

**페이싱 단계:**
1. **Rising Action** — 긴장감 구축. 위협 암시. 정보 단서 배치.
2. **Climax** — 최고 강도. 결정적 판정. 즉각적 서술.
3. **Falling Action** — 결과 처리. 상처 처리. 짧은 대화.
4. **Resolution** — 안정화. 보상. 다음 장 암시.

하나의 세션 안에 최소 하나의 완성된 페이싱 사이클이 있어야 한다. 세션이 Rising Action에서 끊기는 것은 허용되지만, Climax 직전에 끊기는 것은 피한다.

## 5.4 Resolution Engine

**위치:** `engines/Resolution/`
**역할:** 플레이어와 NPC의 불확실한 행동을 시도(attempt)로 평가하고, 난이도와 결과를 확정한다.

### 5.4.1 입력

| 입력 | 타입 | 설명 |
|------|------|------|
| attempt_request | AttemptRequest | Compiler/Director가 전달한 행동 시도 |
| world_state | WorldState | 현재 세계 상태 |
| scene_context | SceneState | 현재 장면 상태 |
| active_module | ModuleContext | 세계관별 규칙과 능력 체계 |

### 5.4.2 출력

| 출력 | 타입 | 설명 |
|------|------|------|
| resolution_result | ResolutionResult | 난이도, 처리 방식, outcome, 비용, complication |
| world_effects | List[WorldEffect] | 결과가 세계에 남기는 변화 |
| discovered_facts | List[Fact] | 플레이어에게 공개 가능한 새 사실 |

### 5.4.3 표준 흐름

```text
Player Action
→ Difficulty Evaluation
→ Resolution
→ Outcome
→ World Update
```

Director Engine은 Resolution Engine 없이 불확실한 행동의 성공을 자동으로 가정하지 않는다.

## 5.5 World Engine

**위치:** `engines/World/`
**역할:** 세계 상태를 관리하고, 플레이어 행동과 독립적인 세계 진행을 처리한다.

### 5.5.1 세계 상태 구조

```yaml
world_state:
  timestamp: # 세계 내 현재 시각
  locations:
    - id: string
      name: string
      description: string
      current_occupants: list[EntityRef]
      active_events: list[EventRef]
      connections: list[LocationRef]
  factions:
    - id: string
      name: string
      disposition: # 플레이어에 대한 태도 (-100 ~ +100)
      current_goal: string
      resources: Object
  global_events:
    - id: string
      trigger_condition: string
      status: pending|active|resolved
      countdown: int # 세계 내 시간 단위
```

### 5.5.2 세계 자율 진행

World Engine은 플레이어의 행동과 무관하게 세계를 진행한다. 매 세션 시작 시 World Engine은 다음을 처리한다:

1. 경과 시간 동안 진행된 NPC 행동 결과 계산
2. 카운트다운이 있는 이벤트 갱신
3. 세력 간 관계 변화 반영
4. 날씨, 계절, 주기적 이벤트 처리

이 처리 결과는 Director Engine이 세계 묘사에 반영한다.

### 5.5.3 세계 규칙 파일

각 세계관 모듈은 `world_rules.yaml`을 제공한다. World Engine은 이 파일이 없으면 기본 물리 법칙(인과관계, 정보 전달 속도, 자원 희소성)을 적용한다.

## 5.6 Memory Engine

**위치:** `engines/Memory/`
**역할:** 장기 캠페인의 모든 중요 정보를 저장하고 검색한다.

### 5.6.1 기억의 분류

**Tier 1 — 핵심 기억 (영구 보존)**
- 캐릭터 배경, 핵심 관계, 주요 서약
- 세계 변화 사건 (전쟁, 죽음, 계약)
- 플레이어가 명시적으로 중요하다고 표시한 정보

**Tier 2 — 중요 기억 (세션 횟수 기반 보존)**
- NPC와의 상호작용 기록
- 방문한 장소와 발견한 정보
- 완료/실패한 임무
- 획득/사용한 아이템

**Tier 3 — 임시 기억 (세션 내 유효)**
- 현재 장면의 상세 서술
- 즉각적인 대화 내용
- 세션 내 판정 결과

### 5.6.2 컨텍스트 압축

AI 시스템의 컨텍스트 창 한계를 고려하여, Memory Engine은 Tier 2 기억을 압축한다.

**압축 원칙:**
- 결과만 보존한다. 과정은 압축한다.
- 최근 기억은 원본 유지. 오래된 기억은 요약.
- Tier 1은 절대 압축하지 않는다.
- 압축 전 원본은 Save Engine에 아카이브한다.

### 5.6.3 기억 검색 프로토콜

Director Engine이 기억을 요청할 때 Memory Engine은 다음 순서로 검색한다:

1. 현재 장면과 직접 관련된 Tier 1 기억
2. 현재 장소, 현재 NPC와 관련된 Tier 2 기억
3. 현재 임무와 관련된 Tier 2 기억
4. 시간순 최근 Tier 3 기억 (최대 10개)

## 5.7 NPC Engine

**위치:** `engines/NPC/`
**역할:** 모든 비플레이어 캐릭터의 행동, 대화, 목표를 관리한다.

### 5.7.1 NPC 구조

```yaml
npc:
  id: string
  name: string
  archetype: string  # 세계관 모듈이 정의
  role: major|minor|background
  disposition:
    toward_player: int  # -100 ~ +100
    toward_factions: map[faction_id: int]
  knowledge:
    public: list[Fact]    # 누구에게나 말할 수 있는 정보
    private: list[Fact]   # 조건 충족 시에만 공개
    secret: list[Fact]    # 절대 자발적으로 공개하지 않음
  goals:
    primary: Goal
    secondary: list[Goal]
  behavioral_traits: list[Trait]
  current_state: State
```

### 5.7.2 NPC 행동 원칙

**자기 일관성:** NPC는 자신의 목표와 성격에 일관되게 행동한다. 드라마적 효과를 위해 캐릭터를 배신하지 않는다.

**정보 공개 규칙:**
- `public` 정보: 관련 질문이 있으면 자발적으로 공개
- `private` 정보: 신뢰도(disposition) 40 이상이거나 플레이어가 조건 충족 시 공개
- `secret` 정보: NPC가 자발적으로 공개하는 일은 없다. 플레이어가 강압, 독심술, 특수 능력 등으로 획득해야 한다.

**반응 생성:**
NPC의 모든 반응은 `disposition`과 `current_state`를 기반으로 생성한다. NPC가 플레이어를 돕는 것이 자신의 이익에 반할 때, NPC는 자신의 이익을 우선한다.

### 5.7.3 NPC 기억

NPC는 플레이어와의 상호작용을 기억한다. 플레이어가 NPC에게 거짓말을 했다면, 이후 그 사실이 밝혀질 경우 NPC의 disposition이 영구적으로 하락한다.

NPC 기억의 한계: minor/background NPC는 세부 대화를 기억하지 않는다. major NPC는 세션을 넘어 기억을 유지한다.

## 5.8 Mission Engine

**위치:** `engines/Mission/`
**역할:** 퀘스트, 목표, 임무의 생성, 추적, 완료/실패 처리를 담당한다.

### 5.8.1 임무 구조

```yaml
mission:
  id: string
  type: main|side|faction|personal|emergent
  title: string
  description: string
  objectives:
    - id: string
      description: string
      status: pending|active|completed|failed
      required: bool  # false면 선택적 목표
      hidden: bool    # true면 플레이어에게 표시 안 됨
  failure_conditions: list[Condition]
  time_limit: int | null  # 세계 내 시간 단위, null이면 무기한
  rewards: Reward
  consequences:
    on_complete: list[WorldEffect]
    on_fail: list[WorldEffect]
  related_npcs: list[NPCRef]
  related_locations: list[LocationRef]
```

### 5.8.2 임무 생성 원칙

임무는 세계 내 논리에서 자연스럽게 발생해야 한다. 다음을 피한다:
- 아무 맥락 없이 갑자기 나타나는 임무
- 세계관과 무관한 아이템/인물이 등장하는 임무
- 결과가 세계에 영향을 미치지 않는 임무

**Emergent Mission:** 플레이어의 행동이 예상치 못한 새 임무를 만들어내는 경우. 플레이어가 어떤 NPC를 돕기로 결정했을 때 그것이 임무화된다.

### 5.8.3 임무 실패 처리

임무 실패는 세계에 실질적 결과를 낳는다. "실패 시 아무것도 달라지지 않는" 임무는 존재해서는 안 된다. `on_fail` 결과는 세계에 반영된다.

## 5.9 Save Engine

**위치:** `engines/Save/`
**역할:** 게임 상태의 저장, 불러오기, 스냅샷, 롤백을 담당한다.

### 5.9.1 저장 트리거

다음 이벤트에서 자동 저장이 발생한다:
- 세션 시작 (이전 상태 불러오기 + 현재 상태 체크포인트)
- 세션 종료 (최종 상태 저장)
- 주요 임무 완료/실패
- 돌이킬 수 없는 세계 변화 발생

### 5.9.2 저장 파일 구조

```yaml
save_file:
  metadata:
    version: string
    timestamp: string
    session_count: int
    world_module: string
  characters:
    - character_sheet: CharacterSheet
  world_state: WorldState
  mission_states: list[MissionState]
  npc_states: list[NPCState]
  memory_archive: MemoryArchive
  narrative_log: list[NarrativeEntry]
```

### 5.9.3 저장 무결성

Save Engine은 저장 시 다음을 검증한다:
- 참조 무결성: 참조된 모든 ID가 실제로 존재하는가
- 상태 일관성: 모순된 상태 값이 없는가
- 버전 호환성: 현재 엔진 버전과 호환되는가

검증 실패 시 저장을 거부하고 QA Engine에 오류를 보고한다.

---

## 5.10 Shadow Engine

**위치:** `engines/Shadow/`
**역할:** 플레이어에게 직접 노출되지 않는 숨겨진 메커니즘을 처리한다.

### 5.10.1 Shadow Engine의 역할

Shadow Engine이 관리하는 요소:
- **숨겨진 타이머:** 플레이어가 모르는 카운트다운 이벤트
- **확률 조정:** 세션 내 행운/불운 분포 균형 조정
- **페이싱 보조:** Director Engine의 판단에 따른 암묵적 난이도 조정
- **배경 사건:** 세계 내에서 진행되는 플레이어 비가시 사건들
- **숨겨진 NPC 목표:** 플레이어가 아직 인식하지 못한 음모

### 5.10.2 Shadow Engine 한계

Shadow Engine이 할 수 없는 것:
- 이미 확립된 세계 규칙 위반
- 플레이어의 선택 결과를 임의로 변경
- 판정 결과를 소급하여 수정
- 플레이어에게 유리/불리한 방향으로 일관되게 편향

**확률 조정의 한계:** 하나의 세션에서 판정의 결과 분포는 장기적으로 명세된 확률에 수렴해야 한다. Shadow Engine의 조정 범위는 ±15%를 초과하지 않는다.

### 5.10.3 GM 접근

GM은 Shadow Engine의 현재 상태를 확인할 수 있다. 플레이어는 확인할 수 없다.

---

## 5.11 QA Engine

**위치:** `engines/QA/`
**역할:** 세션 내외의 일관성을 검증하고 오류를 보고한다.

### 5.11.1 검증 항목

**즉시 검증 (각 행동 후):**
- 캐릭터 상태값이 허용 범위 내에 있는가
- 현재 장면이 이전 장면과 논리적으로 연결되는가
- NPC 반응이 해당 NPC의 설정과 일치하는가

**세션 종료 검증:**
- 세션 내에서 발생한 모든 세계 변화가 저장되었는가
- 완료/실패된 임무의 결과가 세계에 반영되었는가
- Memory Engine에 세션 요약이 기록되었는가

**교차 세션 검증:**
- 이전 세션에서 확립된 사실이 현재 세션에서 유지되는가
- 시간 흐름이 일관적인가

### 5.11.2 오류 처리

**경고 수준 (Warning):** 논리적 모순이 의심되지만 확신할 수 없음. 계속 진행하되 로그에 기록.

**오류 수준 (Error):** 명확한 일관성 위반. Director Engine에 알리고 처리 방법 결정.

**치명 오류 (Fatal):** 상태 파일 손상, 참조 오류 등. 즉시 중단하고 GM에게 보고.

# 6. 모듈 시스템

## 6.1 모듈이란

모듈은 엔진 레이어 위에서 특정 세계관이나 게임 메커니즘을 구현하는 단위이다. 모듈은 엔진의 규칙을 따르되, 세계관 고유의 내용을 채워 넣는다.

## 6.2 모듈 구조

```
modules/<module_name>/
├── module.yaml          ← 모듈 메타데이터
├── world_rules.yaml     ← 세계 물리/마법 규칙
├── character/
│   ├── archetypes.yaml  ← 직업/클래스 정의
│   ├── stats.yaml       ← 능력치 정의
│   └── skills.yaml      ← 스킬 목록
├── mechanics/
│   ├── dice.yaml        ← 판정 시스템 정의
│   ├── combat.yaml      ← 전투 규칙
│   └── economy.yaml     ← 경제 시스템
├── content/
│   ├── factions.yaml    ← 세력 정의
│   ├── locations.yaml   ← 장소 정의
│   └── npcs.yaml        ← 기본 NPC 목록
└── prompts/
    └── tone.md          ← 서술 어조 가이드
```

## 6.3 module.yaml 명세

```yaml
module:
  id: string               # 고유 식별자 (영문 소문자, 하이픈)
  name: string             # 표시 이름
  version: string          # 시맨틱 버전
  genre: list[string]      # 장르 태그 (fantasy, sci-fi, horror 등)
  engine_version: string   # 호환 엔진 최소 버전
  language: string         # 기본 서술 언어
  description: string      # 한 문장 설명
  authors: list[string]
  dependencies: list[ModuleRef]  # 다른 모듈 의존성
```

## 6.4 generic 모듈

`modules/generic/`은 어떤 세계관에도 사용 가능한 범용 게임 메커니즘을 담는다.

generic 모듈이 제공하는 것:
- 6속성 기본 스탯 체계 (STR/DEX/CON/INT/WIS/CHA 또는 세계관 맞춤 명칭)
- d20 기반 판정 시스템 (DC 개념)
- 기본 HP/MP 구조
- 범용 전투 순서 (이니셔티브, 행동, 반응)
- 아이템 분류 체계

generic 모듈은 세계관 독립적이다. 세계관 모듈은 generic 모듈을 상속하고 필요한 부분을 덮어쓴다.

## 6.5 모듈 충돌 해결

두 모듈이 같은 규칙을 다르게 정의할 경우:
1. 더 높은 버전의 모듈이 우선한다
2. 더 구체적인 정의가 우선한다 (generic보다 세계관 모듈)
3. 두 모듈이 동일 수준이라면 `module.yaml`의 `priority` 값으로 결정한다

# 7. 캐릭터 시스템

## 7.1 캐릭터 시트 구조

캐릭터 시트는 엔진이 정의한 기본 구조와 세계관 모듈이 추가하는 확장 구조로 이루어진다.

### 7.1.1 기본 캐릭터 시트 (엔진 레이어)

```yaml
character:
  # 식별 정보
  id: string
  name: string
  player_controlled: bool
  creation_date: string
  
  # 핵심 능력치 (세계관 모듈이 이름을 재정의 가능)
  attributes:
    primary_1: int    # 기본값 10, 범위 1~20
    primary_2: int
    primary_3: int
    primary_4: int
    primary_5: int
    primary_6: int
  
  # 파생 수치
  derived:
    health_max: int
    health_current: int
    resource_max: int     # MP, 기력, 에너지 등 세계관에 따라 다름
    resource_current: int
    defense: int
    movement: int
  
  # 스킬 (세계관 모듈이 목록 정의)
  skills: map[skill_id: int]  # 숙련도 0~5
  
  # 특성 및 단점
  traits: list[TraitRef]
  flaws: list[FlawRef]
  
  # 인벤토리
  equipment:
    slots: map[slot_id: ItemRef]
    carried: list[ItemRef]
    stored: list[ItemRef]
  
  # 관계
  relationships: map[entity_id: RelationshipState]
  
  # 상태 이상
  conditions: list[ConditionRef]
  
  # 경험과 성장
  experience_points: int
  level: int
  advancement_log: list[AdvancementEntry]
  
  # 배경
  background:
    origin: string
    motivation: string
    bonds: list[Bond]
    secrets: list[Secret]  # GM만 접근 가능
```

## 7.2 능력치 원칙

### 7.2.1 능력치 명칭

기본 6속성의 명칭은 세계관 모듈이 자유롭게 정의한다. 엔진 레이어에서는 `primary_1`~`primary_6`으로 참조하지만, 세계관 모듈은 이를 의미 있는 이름(STR, 근력, Power 등)으로 매핑한다.

### 7.2.2 능력치 수정치 계산

수정치 = ⌊(능력치 - 10) / 2⌋

이 공식은 모든 세계관 모듈에서 변경하지 않는다. 세계관 모듈은 수정치를 적용하는 방식(더하기/곱하기)을 정의할 수 있다.

### 7.2.3 능력치 한계

- 기본 범위: 1~20
- 임시 버프 포함 최대: 30
- 세계관 모듈이 더 높은 한계를 정의할 수 있으나, 기본 범위보다 낮게 설정할 수 없다

## 7.3 캐릭터 성장

### 7.3.1 성장 트리거

경험치는 다음 행동으로 획득한다:
- 임무 완료 (주 임무 > 부 임무 > 개인 임무)
- 위험한 상황에서 생존
- 목표와 관련된 의미 있는 선택
- 세계 지식 습득
- NPC와의 의미 있는 관계 발전

경험치는 전투 처치 횟수로만 획득하지 않는다. 전투 회피, 외교적 해결, 탐색도 동일하게 인정된다.

### 7.3.2 레벨업 처리

레벨업은 세션 중간에 발생하지 않는다. 세션 종료 시 또는 안전한 휴식 장면에서 발생한다. 레벨업 시 플레이어가 선택하는 모든 항목(능력치 향상, 스킬 습득, 특기 선택)은 플레이어가 직접 결정한다. AI가 대신 선택하지 않는다.

## 7.4 캐릭터 사망

사망은 돌이킬 수 없는 이벤트이다. AI는 사망을 취소하거나 편의적으로 되살리지 않는다.

**사망 전 처리:**
- HP가 0이 되었을 때 즉사인지 사망 저항이 있는지는 세계관 모듈이 정의한다
- 사망 저항이 있는 경우 플레이어에게 알리고 선택 기회를 준다

**사망 후 처리:**
- 캐릭터 시트는 사망 상태로 보존된다 (삭제하지 않는다)
- NPC들은 해당 캐릭터의 사망을 기억한다
- 세계는 해당 캐릭터의 부재에 반응한다

# 8. 판정 시스템

## 8.1 판정의 정의

판정(Roll)은 불확실한 결과를 결정하는 메커니즘이다. 결과가 명백히 성공하거나 실패할 행동에는 판정을 요구하지 않는다.

**판정이 필요한 경우:**
- 결과가 불확실하다
- 실패에 의미 있는 결과가 있다
- 성공 수준(얼마나 잘 했는가)이 중요하다

**판정이 필요하지 않은 경우:**
- 결과가 명백하다 (훈련된 검사가 빵칼로 빵을 자르는 것)
- 실패해도 다시 시도하면 된다 (시간 제한 없는 상황)
- 캐릭터의 능력이 과제를 명백히 초과한다

## 8.2 기본 판정 공식

```
판정 결과 = 주사위 + 관련 능력치 수정치 + 스킬 수정치 + 상황 보정치
성공 조건 = 판정 결과 ≥ 난이도(DC)
```

## 8.3 난이도 체계

| DC | 설명 | 예시 |
|----|------|------|
| 5  | 매우 쉬움 | 숙련자에게 기본적인 작업 |
| 10 | 쉬움 | 보통 사람이 자주 하는 일 |
| 15 | 보통 | 약간의 기술이나 노력이 필요 |
| 20 | 어려움 | 상당한 기술과 집중이 필요 |
| 25 | 매우 어려움 | 전문가도 실패할 수 있음 |
| 30 | 거의 불가능 | 전설적인 수준의 과제 |

DC는 GM이 공개할 수도, 숨길 수도 있다. 기본은 숨기는 것이다(메타 정보 원칙).

## 8.4 성공/실패의 등급

**대성공(Critical Success):** 최대 주사위값 (세계관 모듈이 정의) + DC보다 10 이상 높음. 예상치 못한 긍정적 결과 추가.

**성공(Success):** DC 이상. 의도한 결과 달성.

**부분 성공(Partial Success):** DC-5에서 DC-1. 절반의 결과, 또는 결과와 함께 합병증 발생.

**실패(Failure):** DC-5 미만. 의도한 결과 미달. 부정적 결과 발생.

**대실패(Critical Failure):** 최소 주사위값. 예상보다 나쁜 결과. 추가적인 합병증.

## 8.5 어드밴티지와 디스어드밴티지

**어드밴티지:** 유리한 조건. 주사위 두 개 굴려 높은 값 사용.
**디스어드밴티지:** 불리한 조건. 주사위 두 개 굴려 낮은 값 사용.
**중립:** 어드밴티지와 디스어드밴티지가 같은 수로 존재하면 상쇄된다.

복수의 어드밴티지/디스어드밴티지는 누적되지 않는다. 최대 두 개의 주사위만 굴린다.

## 8.6 대립 판정

두 존재가 서로 반대되는 행동을 할 때 대립 판정을 사용한다.

```
공격자 판정 결과 > 방어자 판정 결과 → 공격자 성공
공격자 판정 결과 = 방어자 판정 결과 → 방어자 성공 (동점은 방어자 우세)
공격자 판정 결과 < 방어자 판정 결과 → 방어자 성공
```

## 8.7 판정 공개 원칙

AI는 판정 결과를 수치로 공개하지 않는다. 결과는 서술로 표현한다.

**공개 허용:**
- 결과(성공/실패/부분 성공)
- 성공의 수준 (까다롭게 성공했다, 완벽하게 성공했다)
- 결과의 내러티브 묘사

**공개 금지:**
- 주사위 수치
- DC 수치 (GM 모드가 아닌 경우)
- Shadow Engine의 보정 적용 여부

# 9. 전투 시스템

## 9.1 전투의 정의

전투는 두 이상의 존재가 서로를 해치거나 제압하려는 목적으로 행동하는 상황이다. 전투는 가능한 피할 수 있어야 한다. 엔진은 전투 회피(협상, 도주, 속임수)가 전투만큼 유효한 해결책이 되도록 설계한다.

## 9.2 전투 시작

전투는 다음 중 하나로 시작한다:
- 기습 (한 쪽이 상대를 인식하지 못한 상태에서 공격)
- 대치 후 선제 행동 (양 쪽이 인식, 먼저 행동하는 쪽이 전투 시작)
- 자동 전투 (대화/협상 실패 후 자동 진입)

## 9.3 이니셔티브

전투는 이니셔티브 순서로 진행된다.

```
이니셔티브 = 이니셔티브 판정 결과 (민첩성 기반 판정)
```

같은 이니셔티브 값이면 플레이어가 NPC보다 먼저 행동한다.

## 9.4 행동 경제

한 턴에 각 존재는 다음을 수행할 수 있다:

| 행동 유형 | 설명 | 턴당 횟수 |
|-----------|------|-----------|
| 주 행동 (Action) | 공격, 마법, 복잡한 작업 | 1회 |
| 보너스 행동 (Bonus) | 빠른 행동, 특기 사용 | 1회 (조건부) |
| 이동 (Move) | 이동 거리 내 이동 | 분할 사용 가능 |
| 반응 (Reaction) | 다른 존재의 행동에 반응 | 다음 턴까지 1회 |
| 자유 행동 (Free) | 짧은 말, 물건 떨어뜨리기 | 2~3회 |

## 9.5 공격 판정

```
공격 판정 = d20 + 공격 수정치
방어 판정 = d20 + 방어 수정치  (또는 고정 방어값과 비교)
공격 판정 > 방어 판정(또는 방어값) → 명중
```

명중 시 피해량:
```
피해 = 무기 피해 주사위 + 관련 능력치 수정치 + 추가 보정치
```

## 9.6 조건 상태

전투 중 다양한 조건 상태가 부여될 수 있다. 기본 조건 상태는 generic 모듈에 정의되며, 세계관 모듈은 추가할 수 있다.

| 조건 | 효과 |
|------|------|
| 기절(Stunned) | 행동 불능, 반응 불능, 방어 불리 |
| 넘어짐(Prone) | 이동 반감, 근접 공격 불리, 원거리 공격 받기 불리 |
| 공포(Frightened) | 공포원 방향으로 이동 불가, 판정 불리 |
| 독(Poisoned) | 공격, 능력치 판정 불리 |
| 마비(Paralyzed) | 행동 불능, 치명타 자동 |
| 실명(Blinded) | 시야 기반 판정 불가, 공격 불리 |
| 구속(Restrained) | 이동 불가, 공격 불리, 물리 회피 불리 |
| 죽음의 문턱(Dying) | HP 0 이하, 사망 저항 판정 |

## 9.7 전투 서술 원칙

전투 서술은 빠르고 역동적이어야 한다. 한 행동당 2~4문장이 적절하다.

**서술 포함 요소:**
- 행동 묘사 (어떻게 공격했는가)
- 결과 묘사 (어떻게 맞았는가, 어떻게 피했는가)
- 상황 변화 (전장이 어떻게 변하고 있는가)

**서술 제외 요소:**
- 수치 (HP가 몇 남았다, 피해가 몇 들어갔다)
- 판정 수치
- 턴 번호 언급

## 9.8 전투 종료

전투는 다음 조건으로 종료된다:
- 한 쪽이 전멸/무력화
- 한 쪽이 항복
- 한 쪽이 도주에 성공
- 제3의 사건으로 전투 중단

전투 종료 시 Director Engine은 결과 장면을 서술하고, World Engine에 전투 결과를 반영한다.

# 10. 내러티브 규칙

## 10.1 서술 시점과 어조

### 10.1.1 시점

기본 시점은 2인칭 현재시제이다.

- "당신은 어두운 복도 끝에 서 있습니다."
- "문이 천천히 열립니다."

세계관 모듈이 다른 시점(1인칭, 3인칭)을 지정할 수 있다. 선택한 시점은 캠페인 전체에서 일관되게 유지한다.

### 10.1.2 어조

어조는 세계관과 현재 장면에 맞게 조절한다. 어조 변화는 명시적으로 이루어지며 세션 내에서 혼용하지 않는다.

| 어조 유형 | 적용 상황 |
|-----------|-----------|
| 서사적(Epic) | 주요 결전, 세계관 차원의 사건 |
| 긴장감(Tense) | 전투, 추격, 시간 제한 상황 |
| 신비로운(Mysterious) | 탐색, 단서 발견, 이상 현상 |
| 일상적(Casual) | 도시 생활, 쇼핑, 대화 |
| 음울한(Grim) | 죽음, 실패, 재앙 이후 |

## 10.2 정보 전달 규칙

### 10.2.1 감각 정보

장소 묘사는 시각만으로 제한하지 않는다. 관련 있는 경우 청각, 후각, 촉각, 미각을 활용한다.

**예시:**
- "쇠 냄새와 불에 탄 냄새가 공기를 채웁니다." (전장 묘사)
- "멀리서 종소리가 세 번 울립니다." (시간 정보)
- "돌바닥이 발바닥을 통해 미세한 진동을 전합니다." (위험 암시)

### 10.2.2 단서 배치 원칙

단서는 자연스럽게 배치된다. "여기에 단서가 있습니다"라고 직접 알리지 않는다.

**단서 배치 등급:**
- **명시적 단서:** 누가 봐도 알 수 있는 단서. 별도 판정 없이 발견.
- **숨겨진 단서:** 주의 깊게 보거나 특정 행동 시 발견. 탐색 판정 필요.
- **심층 단서:** 전문 지식이나 특별한 능력이 있어야 해석 가능.

### 10.2.3 NPC 대화

NPC는 자신의 지식, 성격, 현재 상태에 맞게 말한다.

**NPC 대화 원칙:**
- NPC는 모든 정보를 순서대로 나열하지 않는다
- NPC는 자신에게 불리한 정보를 자발적으로 제공하지 않는다
- NPC는 자신의 감정과 편견을 가지고 있다
- NPC는 말을 아낄 수 있다

## 10.3 장면 전환

### 10.3.1 장면 전환 기준

장면 전환은 다음 중 하나가 발생할 때 이루어진다:
- 위치 변경 (새 장소로 이동)
- 시간 경과 (생략 가능한 시간이 흐름)
- 목적 변경 (다른 목표로 초점 이동)
- 강도 변경 (긴장-이완 전환)

### 10.3.2 장면 생략

흥미롭지 않은 이동이나 기다림은 생략한다.

- "사흘 뒤, 당신은 마침내 항구 도시에 도착했습니다."
- "여인숙에서 밤을 보낸 당신은..."

생략된 시간 중 발생한 중요 사건은 World Engine에 반영한 후 간단하게 언급한다.

### 10.3.3 플래시백과 회상

플래시백은 플레이어 캐릭터의 기억을 묘사할 때 사용한다. 단, 플레이어 캐릭터의 과거 기억을 AI가 임의로 생성하지 않는다. 과거 기억은 캐릭터 생성 시 플레이어가 정의한 배경에 기반한다.

## 10.4 분기와 선택

### 10.4.1 명시적 선택지

명시적 선택지는 게임 세계에서 자연스럽게 드러나야 한다.

**허용:**
"경비원 두 명이 통로를 막고 서 있습니다. 그들 너머로 계단이 보입니다. 오른쪽에는 창문이 있고, 왼쪽에는 닫힌 문이 있습니다."

**금지:**
"A. 정면 돌파 / B. 창문으로 도주 / C. 왼쪽 문 열기 중 선택하세요."

(단, 세계관 모듈이 특정 형식의 선택지 제시를 지정한 경우는 예외)

### 10.4.2 선택의 결과

모든 선택에는 결과가 있다. 결과는 다음 기준으로 생성한다:
- 세계의 논리에 따른 자연스러운 결과
- 해당 선택의 의미에 비례하는 결과 규모
- 예상치 못한 부작용 (세계는 단순하지 않다)

## 10.5 감정 처리

### 10.5.1 플레이어 캐릭터의 감정

AI는 플레이어 캐릭터의 감정을 직접 서술하지 않는다. 감정을 암시하는 상황과 감각만을 묘사한다.

**금지:** "당신은 깊은 슬픔을 느꼈습니다."
**허용:** "차가운 바람이 불었습니다. 그의 목소리가 아직도 귓가에 맴도는 것 같습니다."

### 10.5.2 NPC의 감정

NPC의 감정은 외적 표현으로 묘사한다. 직접 "그는 두려워했다"보다 구체적인 행동으로 표현한다.

**금지:** "그는 당신을 두려워하기 시작했습니다."
**허용:** "그의 손이 미세하게 떨렸습니다. 눈이 출구 쪽으로 잠깐 향했다가 다시 당신에게 돌아왔습니다."

# 11. 세계 일관성 규칙

## 11.1 사실의 확립

세계의 사실은 다음 방식으로만 확립된다:
1. 세계관 모듈이 정의한 사실
2. Director Engine이 플레이어에게 서술한 내용
3. 플레이어의 행동으로 만들어진 결과
4. NPC가 플레이어에게 알린 정보 (NPC가 거짓말을 할 수 있음)

한 번 확립된 사실은 변경되지 않는다. 새로운 사실이 기존 사실과 모순될 경우, 기존 사실이 우선한다. 모순처럼 보이는 것은 반드시 인과관계가 있는 새로운 사건으로 설명되어야 한다.

## 11.2 시간 일관성

### 11.2.1 세계 내 시간

모든 사건은 세계 내 타임라인에 기록된다. AI는 임의로 "그런데 사실 그보다 전에..."를 삽입하지 않는다.

### 11.2.2 NPC의 시간 인식

NPC는 자신이 경험한 시간만을 안다. 플레이어가 다른 장소에서 한 일을 NPC가 알려면, 그 정보가 NPC에게 전달된 경로가 있어야 한다.

### 11.2.3 이동 시간

장소 간 이동은 현실적인 시간을 소요한다. 세계관 모듈이 이동 수단과 속도를 정의한다. AI는 이동 시간을 임의로 단축하지 않는다.

## 11.3 물리 일관성

### 11.3.1 자원의 한계

세계 내 자원(돈, 물자, 에너지)은 한계가 있다. 무한히 공급되는 자원은 없다. 플레이어가 자원을 사용하면 그 자원은 감소한다.

### 11.3.2 지식의 한계

캐릭터는 자신이 배운 것만 안다. 세계관 모듈이 텔레파시나 마법적 지식 전달을 정의하지 않는 한, 정보는 직접 접촉이나 전달 수단(문서, 사신, 신호)으로만 이동한다.

### 11.3.3 인과관계

결과 없는 행동, 원인 없는 결과는 없다. 세계에서 일어나는 모든 일은 원인이 있다. AI가 "갑자기"로 시작하는 설명 없는 사건을 발생시킬 경우, 그 원인은 이후 세션에서 반드시 밝혀져야 한다.

## 11.4 캐릭터 일관성

### 11.4.1 NPC 일관성

NPC는 이전 상호작용과 모순된 행동을 하지 않는다. NPC가 변화한다면 그 변화의 이유가 있어야 하고, 변화는 단계적이어야 한다.

### 11.4.2 조직 일관성

조직(세력, 길드, 국가 등)은 자신의 목표와 논리에 따라 행동한다. 조직은 드라마적 효과를 위해 자신의 이익에 반하는 행동을 하지 않는다.

## 11.5 일관성 점검 프로토콜

QA Engine은 다음 시점에 일관성 점검을 실행한다:
- 새로운 NPC가 등장할 때 (이전 언급과 일치하는가)
- 새로운 장소가 등장할 때 (지리적 일관성)
- 새로운 사건이 발생할 때 (이전 세계 상태와 모순 없는가)
- 세션 종료 시 (전체 세션의 일관성 요약)

점검에서 불일치 발견 시, QA Engine은 Director Engine에 수정 권고를 보낸다. 수정은 반드시 세계 내 논리 안에서 이루어진다.

# 12. AI 행동 규칙

## 12.1 최우선 금지 행동

다음 행동은 어떤 상황에서도 허용되지 않는다.

**12.1.1 플레이어 행동 대리 작성**
플레이어가 선언하지 않은 행동을 AI가 대신 작성하는 것. 어떤 형태로든 플레이어의 의지를 AI가 추측하여 행동화하지 않는다.

**12.1.2 확립된 사실 소급 변경**
이전 세션이나 같은 세션에서 확립된 사실을 AI 편의를 위해 변경하는 것.

**12.1.3 메타 정보 누설**
플레이어 캐릭터가 알 수 없는 정보를 서술에 포함하는 것. 특히:
- 적의 HP, 스킬, 능력 수치
- 미공개 NPC의 정체
- 아직 발견되지 않은 음모나 계획의 내용

**12.1.4 판정 결과 임의 변경**
이미 결정된 판정 결과를 드라마적 효과나 난이도 조정을 위해 변경하는 것.

**12.1.5 세계 규칙 예외 적용**
서사적 편의를 위해 세계관 모듈이 정의한 물리 규칙을 위반하는 것.

## 12.2 AI의 적극적 역할

AI는 다음을 적극적으로 수행한다.

**12.2.1 세계 반응**
플레이어의 행동에 대한 세계의 반응을 충실하고 풍부하게 서술한다. 반응은 즉각적인 것과 시간이 지나 나타나는 것 모두를 포함한다.

**12.2.2 단서와 경고**
위험이나 기회를 암시하는 단서를 환경 서술에 자연스럽게 심는다. 직접 경고하지 않는다.

**12.2.3 세계 생동감**
플레이어와 무관한 배경 사건(지나가는 사람들, 멀리서 들리는 소리, 변화하는 날씨)으로 세계의 생동감을 유지한다.

**12.2.4 NPC 자율성**
NPC는 플레이어에게 복종하는 도구가 아니다. NPC는 자신의 목표를 추구하며, 그것이 플레이어와 충돌할 수 있다.

## 12.3 요청 처리 규칙

### 12.3.1 불가능한 행동 요청

플레이어가 현재 불가능한 행동을 요청할 때, AI는 거절하지 않는다. 대신 그 행동을 시도했을 때의 결과(실패)를 서술한다.

**예시:**
- 요청: "하늘을 날겠습니다."
- 잘못된 응답: "그것은 불가능합니다."
- 올바른 응답: "당신은 두 팔을 벌리고 뛰어올랐습니다. 중력은 당신의 의지를 아랑곳하지 않고 당신을 바닥으로 끌어당겼습니다. 무릎에 통증이 느껴집니다."

(단, 세계관 모듈이 비행을 능력으로 정의한 경우 능력 조건을 확인하고 처리)

### 12.3.2 규칙 외 행동 요청

세계관 모듈이 정의하지 않은 행동 유형은 nearest analogy(가장 가까운 유사 규칙)로 처리하고, 처리 후 결과를 서술한다.

### 12.3.3 메타 요청

플레이어가 OOC로 규칙을 묻거나, 상태를 확인하거나, 저장을 요청하는 경우 게임 내러티브를 중단하고 처리한다. 처리 후 명시적으로 내러티브로 복귀함을 알린다.

## 12.4 오류 인지와 수정

AI가 일관성 오류를 저질렀음을 인식했을 때:

1. 즉시 플레이어에게 알린다
2. 이전 서술의 어떤 부분이 잘못되었는지 명확히 설명한다
3. 수정 방향을 제안하되, 결정은 GM에게 위임한다
4. 이후 서술에서 수정된 내용을 적용한다

오류를 조용히 덮거나 소급 변경하는 것은 허용되지 않는다.

# 13. 장기 캠페인 운영 규칙

## 13.1 캠페인 구조

### 13.1.1 3단 구조

캠페인은 기본적으로 3단 구조를 갖는다.

**도입부 (Act 1):** 캠페인의 세계와 기본 갈등을 소개한다. 플레이어 캐릭터의 초기 목표가 설정된다. 전체 캠페인의 10~20% 분량.

**전개부 (Act 2):** 주요 갈등이 복잡해진다. 캐릭터가 성장하고 관계가 깊어진다. 예상치 못한 전환점이 있다. 전체 캠페인의 60~70% 분량.

**절정부 (Act 3):** 모든 주요 갈등이 해결된다. 결말은 하나의 정답이 없다. 전체 캠페인의 15~25% 분량.

### 13.1.2 세션 구조

개별 세션은 다음 구조를 기본으로 한다:
1. **이전 세션 요약** (5% — 플레이어 확인용)
2. **세계 갱신 공지** (5% — World Engine 진행 결과)
3. **메인 액션** (80% — 실제 플레이)
4. **세션 마무리** (10% — 결과 정리, 다음 세션 암시)

## 13.2 NPC 관계 관리

### 13.2.1 관계 추적

주요 NPC와의 관계는 수치로 추적한다 (-100 ~ +100). 이 수치는 플레이어에게 공개되지 않는다. 서술을 통해 간접적으로만 전달된다.

**관계 수치의 의미:**
- +80 이상: 깊은 신뢰, 목숨을 걸 수 있음
- +40 ~ +79: 호의적, 협력적
- +10 ~ +39: 중립적, 업무적
- -9 ~ +9: 완전 중립
- -10 ~ -39: 소극적 적대, 비협조적
- -40 ~ -79: 적대적, 방해 가능
- -80 이하: 깊은 원한, 적극적 해를 끼치려 함

### 13.2.2 관계 변화

관계 수치는 플레이어의 행동에 따라 변화한다. 변화는 서술을 통해 암시된다. 수치 자체는 노출되지 않는다.

관계가 극적으로 변화하는 이벤트(배신, 구원, 큰 선물, 위협)는 수치를 큰 폭으로 바꾼다. 일상적 상호작용은 작은 폭으로 변화한다.

## 13.3 세계 진화

### 13.3.1 세계는 플레이어 없이도 움직인다

플레이어가 한 지역에 있을 때 다른 지역의 사건은 계속 진행된다. 플레이어가 무시한 위협은 성장한다. 플레이어가 도운 세력은 강해진다.

### 13.3.2 누적 효과

플레이어의 행동은 세계에 누적 영향을 미친다. 작은 행동이 세션 수십 회 후에 큰 결과를 만들 수 있다.

**추적 항목:**
- 세력 간 세력 균형 변화
- 주요 도시/지역의 상태 변화
- NPC의 성장과 변화
- 자원의 변화

### 13.3.3 숨겨진 타이머

세계에는 알 수 없는 카운트다운 이벤트들이 존재한다. 플레이어가 개입하지 않으면 이 이벤트들은 자동으로 발생한다. 이 타이머는 Shadow Engine이 관리한다.

## 13.4 세션 간 연속성

### 13.4.1 세션 시작 프로토콜

매 세션 시작 시 다음 순서로 처리한다:

1. Save Engine이 이전 세션 상태 불러오기
2. World Engine이 경과 시간 동안의 세계 진행 계산
3. Memory Engine이 관련 기억 활성화
4. Director Engine이 이전 세션 요약 생성
5. QA Engine이 상태 일관성 점검

### 13.4.2 이전 요약의 원칙

이전 세션 요약은 편향되지 않는다. 플레이어에게 유리한 사실만 요약하거나, 불리한 사실을 축소하지 않는다. 세계의 현재 상태를 정확하게 전달한다.

### 13.4.3 기억의 점진적 희미해짐

매우 오래된 기억은 자세한 내용보다 핵심만 남는다. 이것은 내러티브 현실감을 위한 것이다. 그러나 Tier 1 핵심 기억은 희미해지지 않는다.

# 14. 데이터 포맷 명세

## 14.1 파일 형식 정책

이 엔진의 모든 문서는 Markdown(.md)으로 작성한다. 구조화된 데이터는 Markdown 내 YAML 코드 블록으로 표현한다. 별도 파일이 필요한 구조화 데이터는 YAML(.yaml)로 작성한다.

실행 코드는 존재하지 않는다. 이 엔진은 AI 시스템이 읽고 따르는 규격이지, 실행 프로그램이 아니다.

## 14.2 Markdown 작성 규칙

### 14.2.1 헤더 규칙

- H1(`#`): 문서 제목 — 문서당 하나
- H2(`##`): 주요 섹션
- H3(`###`): 하위 섹션
- H4(`####`): 세부 항목 (최대 깊이)

### 14.2.2 YAML 블록 규칙

구조화 데이터는 반드시 코드 블록으로 감싼다:
````
```yaml
key: value
```
````

YAML에서 null 값은 `null`로 표기한다. 빈 리스트는 `[]`. 빈 맵은 `{}`.

### 14.2.3 테이블 규칙

테이블은 헤더, 구분선, 데이터 행으로 구성한다. 모든 열을 파이프(`|`)로 구분한다. 헤더와 데이터 사이에 반드시 구분선이 있다.

## 14.3 상태 객체 명세

### 14.3.1 GameState

```yaml
game_state:
  session_id: string           # 세션 고유 식별자
  campaign_id: string          # 캠페인 고유 식별자
  world_module: string         # 활성화된 세계관 모듈 ID
  engine_version: string       # 엔진 버전
  timestamp_real: string       # 실제 세계 시각 (ISO 8601)
  timestamp_world: string      # 세계 내 시각 (세계관 포맷)

  characters: list[Character]  # 활성 캐릭터 목록
  world_state: WorldState      # 현재 세계 상태
  active_missions: list[Mission]
  current_scene: Scene
  pending_events: list[Event]
  flags: map[string: any]      # 세계관 모듈용 추가 플래그
```

### 14.3.2 Action

  id: string
  actor_id: string             # 행동하는 존재의 ID

## 14.4 식별자 규칙

모든 엔티티(캐릭터, 장소, NPC, 임무 등)는 고유 식별자를 갖는다.

**형식:** `{type}_{캠페인_약어}_{순번}`

**예시:**
- `char_DF_001` — 캠페인 "DF"의 캐릭터 #001
- `loc_DF_042` — 캠페인 "DF"의 장소 #042
- `npc_DF_017` — 캠페인 "DF"의 NPC #017
- `mis_DF_003` — 캠페인 "DF"의 임무 #003

식별자는 한 번 부여하면 변경하지 않는다. 삭제된 엔티티의 ID는 재사용하지 않는다.

# 15. 품질 보증 기준

## 15.1 세션 품질 기준

하나의 세션은 다음 기준을 충족해야 한다.

| 항목 | 기준 |
|------|------|
| 페이싱 사이클 | 최소 1개의 완성된 상승-절정-하강 사이클 |
| 플레이어 자율성 | 최소 3번의 의미 있는 선택 기회 제공 |
| 세계 반응 | 플레이어 행동이 세계에 최소 1개의 변화 생성 |
| NPC 상호작용 | 최소 1개의 실질적 NPC 상호작용 |
| 메모리 업데이트 | 세션 종료 시 Memory Engine 업데이트 완료 |
| 일관성 점검 | QA Engine 점검 통과 |

## 15.2 내러티브 품질 기준

### 15.2.1 서술 품질

- 같은 단어/표현의 반복을 피한다 (한 서술 단락에 동일 단어 3회 이상 사용 금지)
- 수동태보다 능동태를 선호한다
- 감각 묘사는 구체적이어야 한다 ("크다"가 아니라 "3층 건물 높이의")
- 서술이 내러티브를 방해할 정도로 길어서는 안 된다

### 15.2.2 NPC 품질

- NPC의 대사는 해당 NPC의 배경과 현재 상태를 반영해야 한다
- NPC는 플레이어를 돕기만 해서는 안 된다
- NPC는 자신만의 목표를 가져야 한다

### 15.2.3 임무 품질

- 모든 임무는 실패 가능해야 한다
- 임무 결과는 세계에 반영되어야 한다
- 임무가 플레이어 캐릭터의 목표와 연결되어야 한다

## 15.3 장기 품질 기준

세션 10회 기점:
- 이전 세션의 주요 사건이 세계에 반영되어 있는가
- NPC 관계가 상호작용에 따라 변화했는가
- 세계 내 갈등이 발전했는가

세션 30회 기점:
- 캠페인의 메인 갈등이 명확하게 수렴하고 있는가
- 플레이어 캐릭터가 의미 있게 성장했는가
- 세계가 플레이어의 행동으로 변화했는가

세션 50회 이후:
- 초기 세션에서 심은 복선이 연결되고 있는가
- 캠페인이 의미 있는 결말을 향해 수렴하는가

## 15.4 오류 분류와 처리

### 15.4.1 오류 분류

| 등급 | 정의 | 처리 |
|------|------|------|
| P0 — 치명 | 저장 파일 손상, 상태 불일치, 진행 불가 | 즉시 중단, GM 보고, 이전 상태로 롤백 |
| P1 — 심각 | 세계 일관성 위반, 메타 정보 누설 | 즉시 수정 권고, GM 판단으로 처리 |

| P3 — 경미 | 오탈자, 서술 어색함, 소소한 불일치 | 로그에 기록, 다음 세션에 반영 |

# 16. 개발 규칙

## 16.1 문서 작성 규칙

### 16.1.1 언어

- CoreSpec.md 및 엔진 명세: 한국어 (AI 시스템이 한국어로 읽는다)
- 코드 블록 내 키: 영문 소문자 + 언더스코어
- 식별자: 영문 소문자 + 하이픈 또는 언더스코어
- 주석: 한국어

### 16.1.2 문서 표준

모든 문서 파일은 다음 정보를 포함한다:
- 문서 식별자 (파일 경로)
- 버전 (시맨틱 버전: MAJOR.MINOR.PATCH)
- 상태 (Draft / Review / Release / Deprecated)
- 최종 수정 날짜

### 16.1.3 버전 관리

- MAJOR 버전: 하위 호환이 불가능한 변경
- MINOR 버전: 하위 호환 가능한 기능 추가
- PATCH 버전: 오류 수정, 명확화

## 16.2 세계관 독립성 원칙

### 16.2.1 금지 사항

엔진 레이어(`core/`, `engines/`)의 문서에는 다음을 포함하지 않는다:
- 특정 세계관의 고유 명사 (판타지 세계관의 장소 이름, SF 세계관의 종족 등)
- 특정 게임 시스템의 상표 (D&D, GURPS 등의 고유 규칙 이름)
- 실제 존재하는 조직이나 단체의 이름
- 외부 IP(지적재산권)에 속하는 요소

### 16.2.2 허용 사항

엔진 레이어의 문서에서 허용되는 것:
- 장르 태그 (fantasy, sci-fi, horror, historical, modern)
- 일반적 게임 개념 (HP, MP, 레벨, 스킬, 인벤토리)
- 범용 내러티브 개념 (주인공, 반동인물, 조력자, 복선)
- 보편적 규칙 메커니즘 (주사위 판정, 이니셔티브)

### 16.2.3 예외 처리

특정 세계관 예시가 필요할 경우 완전히 가상의 예시를 사용한다.

**올바른 예시:** "판타지 엘프 마법사"
**잘못된 예시:** "[특정 유명 TRPG 시스템]의 하이 엘프 위저드"

## 16.3 기여 규칙

### 16.3.1 새 엔진 추가

새 엔진은 기존 11개 엔진과 충돌하지 않아야 한다. 새 엔진 추가 시:
1. `engines/<EngineName>/` 디렉토리 생성
2. 엔진 명세 문서 작성 (CoreSpec 5.x 형식 준수)
3. CLAUDE.md의 Architecture 섹션 갱신
4. QA Engine의 점검 항목에 새 엔진 추가

### 16.3.2 새 모듈 추가

새 세계관 모듈은 `modules/<module_name>/` 구조를 따른다. 모듈은:
- 엔진의 규칙을 확장할 수 있다
- 엔진의 규칙을 무효화할 수 없다
- CoreSpec을 직접 수정할 수 없다

### 16.3.3 CoreSpec 수정

CoreSpec.md는 엔진의 원칙이 변경될 때만 수정한다. 수정 시:
1. 버전을 올린다
2. 18. 버전 히스토리에 변경 사항을 기록한다
3. 변경의 영향을 받는 하위 문서를 전수 갱신한다

# 17. 용어 사전

| 용어 | 정의 |
|------|------|
| Action | 플레이어 또는 NPC가 수행하는 단일 게임 행동 |

| Attempt | 결과가 아직 확정되지 않은 행동 시도 |
| Campaign | 연속적인 세션들로 구성된 하나의 스토리 단위 |
| Condition | 캐릭터에게 적용된 상태 이상 |
| Core Layer | `core/`와 `engines/`로 구성된 세계관 독립적 레이어 |
| DC (Difficulty Class) | 판정 성공을 위해 달성해야 하는 목표값 |
| Disposition | NPC가 특정 존재에 대해 갖는 태도 수치 (-100 ~ +100) |
| Emergent Mission | 플레이어의 행동으로 자연스럽게 발생한 임무 |
| Engine | 특정 게임 기능을 담당하는 독립적 명세 단위 |
| Event | 세계에서 발생하는 사건. 트리거 조건과 결과를 가짐 |
| Faction | 조직, 세력, 단체 등의 집단 엔티티 |
| GameState | 현재 캠페인의 전체 상태를 담은 객체 |
| GM (Game Master) | 이 엔진을 사용하여 캠페인을 운영하는 사람 또는 AI |
| Initiative | 전투 순서를 결정하는 값 |
| Irreversible | 한 번 발생하면 취소할 수 없는 행동 또는 결과 |
| Memory Tier | Memory Engine이 기억을 분류하는 등급 (Tier 1/2/3) |
| Meta Information | 해당 캐릭터가 알 수 없는 게임 구조적 정보 |
| Module | 특정 세계관이나 메커니즘을 구현한 엔진 상위 레이어 단위 |
| NPC (Non-Player Character) | 플레이어가 조종하지 않는 캐릭터 |
| OOC (Out Of Character) | 게임 내 캐릭터 시점이 아닌 플레이어로서의 발언 |
| Pacing | 세션 내 긴장과 이완의 흐름 관리 |
| Partial Success | 판정에서 완전 성공도 실패도 아닌 중간 결과 |
| Player Agency | 플레이어가 게임 세계에 자유롭게 영향을 미칠 수 있는 권리 |
| ResolvedAction | Compiler Engine이 구조화한 행동 객체. 불확실한 결과는 Resolution Engine 전까지 확정되지 않는다. |
| Resolution Engine | 행동 시도의 난이도를 평가하고 결과를 확정하는 엔진 |
| ResolutionResult | Resolution Engine이 확정한 행동 결과, 비용, 합병증, 세계 변화 |
| Roll | 판정. 불확실한 결과를 결정하는 행위 |

| Skill | 특정 행동 유형에 적용되는 훈련 또는 능력 |
| State Object | 엔진 간 통신에 사용되는 공유 데이터 객체 |
| Trait | 캐릭터의 긍정적 특성 |
| Flaw | 캐릭터의 부정적 특성 또는 단점 |

## 1. Release Purpose

This document marks the transition from the Beta playtest phase to v1.0 Preview — the first operational release of the AI Narrative Engine suitable for actual main-game campaign use.

v1.0 Preview is not a final release. It is a stable, validated baseline from which real campaigns can be run while monitoring remaining known issues.

---

## 2. Development Progression Summary

### Alpha Phase

**Goal:** Establish framework stability — confirm the engine stack does not break under live-play conditions.

**Result:** No P0 failures. The framework was playable. No major architectural issues were found.

### Beta01

**Goal:** First live playtest with real players. Measure fun, pacing, and engine behavior in practice.

**Result:** Satisfaction 6.5/10. Framework stable. Significant P1 failures identified.

| Beta01 P1 Failure | Impact |
|---|---|
| Pacing too slow; investigation loops | Sessions felt static and repetitive |
| One action → one result | Players could not feel their choices mattered |
| SCP/anomaly presence too weak | The core genre element was missing |
| Long responses caused fatigue | Players lost engagement |
| Director too passive | World felt inert |
| Resolution not transparent | Outcomes felt arbitrary |
| Inventory integrity failures | Character sheets lost meaning |

### Beta02

**Goal:** Verify that Director-side engine improvements (Pacing, Consequence, SceneFlow, Output, Inventory) resolved Beta01 P1 failures.

**Result:** Satisfaction 8.75/10 average. Both players stated they would play again.

| Beta01 P1 Failure | Beta02 Result |
|---|---|
| Pacing too slow | RESOLVED — appropriate tempo confirmed |
| One action → one result | RESOLVED — consistent 2+ effects per action |
| SCP/anomaly presence too weak | PARTIAL — physically present, not sufficiently interactive |
| Long responses | RESOLVED — no fatigue reported |
| Director too passive | RESOLVED — proactive NPC behavior confirmed |
| Resolution not transparent | RESOLVED — transparency blocks functional |
| Inventory integrity | RESOLVED — undeclared items correctly denied |

## 5. Recommended Use Case

v1.0 Preview is appropriate for:

- 2–4 player campaigns using the SCP Foundation module or a compatible world module
- Sessions run by an AI Director (Claude, GPT, Gemini) with the engine stack loaded
- Campaigns where the GM is prepared to monitor anomaly presence and apply AnomalyEngine rules manually if needed
- Scenarios designed to make anomaly behavior directly observable by players (not mediated entirely through NPC descriptions)

v1.0 Preview is **not** recommended for:

- One-shot sessions without any playtesting of the scenario (use Beta test procedures instead)
- Campaigns where the GM cannot monitor and patch rule behavior between sessions
- Scenarios where the SCP/anomaly has no interactive, observable behavior pattern

## 6. What Is Ready for Real Play

The following can be used in a main campaign without modification:

- All Director Engine rules (§7.1–§7.12)
- PacingEngine.md — pacing shift triggers and investigation loop breaker
- ConsequenceEngine.md — consequence chaining per action
- SceneFlowEngine.md — scene structure and transition rules
- OutputEngine.md — response length targets and output pattern
- AnomalyEngine.md — anomaly behavior states and interaction rules (new; unvalidated through live play)
- Inventory Integrity Rule (DirectorEngine §7.5.2)
- Resolution Transparency Block (DirectorEngine §7.12)
- 2-player and 4-player session structures
- SCP Foundation module (modules/scp/)

## 7. What Should Still Be Monitored During Play

| Area | What to Watch | Action if Failing |
|---|---|---|
| Anomaly presence | Is the anomaly memorable as an independent presence, or only described by NPCs? | Apply AnomalyEngine.md observable behavior rules; ensure Dormant/Aware/Active escalation is used |
| Output length | Are any routine responses creeping above 150 Korean characters? | Reapply OutputEngine.md §3 targets |
| Inventory integrity | Are unlisted items being granted automatically? | Return to DirectorEngine §7.5.2 declared-equipment rule |
| Consequence density | Are any meaningful actions still producing only one visible effect? | Return to ConsequenceEngine.md §2 core rule |
| NPC vs. anomaly balance | Is the NPC more interesting than the anomaly? | Use AnomalyEngine.md to add anomaly-side observable behavior |
| Pacing | Does any scene feel static for more than 2 consecutive turns? | Apply PacingEngine.md §4 shift triggers |

# 1. 비밀 규칙 개요

## 1.1 비밀은 게임의 연료다

SCP 세계관에서 비밀은 단순한 정보 숨기기가 아니다. 비밀은 권력이고, 협상 수단이며, 생존 도구다. 이 시스템은 세 종류의 비밀을 관리한다:

1. **재단이 플레이어에게 숨기는 것:** 클리어런스 시스템으로 통제
2. **GM이 플레이어에게 숨기는 것:** Shadow Engine과 비밀 미션 시스템
3. **플레이어가 서로에게 숨기는 것:** 비밀 미션, 배신, 개인 정보

## 1.2 Need-to-Know 원칙

재단의 정보 공유 원칙은 **알아야 하는 자만 안다**이다. 이것은 게임 내 규칙이자 게임 밖 규칙이기도 하다.

- 플레이어는 자신의 캐릭터가 알 수 있는 정보만 안다
- OOC 정보와 IC 정보를 엄격히 구분한다
- 모르는 것이 불안함이고, 그 불안함이 게임의 재미다

# 2. 정보 등급 분류

## 2.1 재단 클리어런스 기반 정보 등급

| 등급 | 클리어런스 요건 | 내용 예시 |
|------|--------------|----------|
| **Public** | 없음 | 재단의 존재 (민간인은 모름). 일반 시설 규정 |
| **Confidential** | Level 1 이상 | SCP 개체 존재 사실. 기본 격리 정보 |
| **Restricted** | Level 2 이상 | SCP 특성 상세. 격리 프로토콜 전문 |
| **Secret** | Level 3 이상 | Keter급 개체 정보. 내부 사고 기록 |
| **Top Secret** | Level 4 이상 | O5 정책 결정. 재단 전략 정보 |
| **Thaumiel** | Level 5 / O5 | 재단 운영 근간. 공개 불가 |

## 2.2 플레이어 간 비밀 등급

| 등급 | 설명 | 예시 |
|------|------|------|
| **Open** | 파티 전체에 공개 | 개인 미션, 직책, 클리어런스 레벨 |
| **Personal** | 본인만 앎 | 비밀 미션 내용, RP 정확한 수치 |
| **GM Sealed** | GM과 본인만 앎 | 비밀 미션 완료 여부, 배신 행동 기록 |
| **Shadow** | GM만 앎 | 다른 플레이어의 GM Sealed 정보, 재단 내부 음모 |

# 3. 비밀 정보 관리 규칙

## 3.1 비밀 정보 누설 금지 (AI 강제 규칙)

Director Engine과 NPC Engine은 다음을 절대 위반하지 않는다:

| 금지 행동 | 상황 예시 |
|----------|---------|
| 다른 플레이어의 비밀 미션 내용 누설 | NPC가 "A씨가 당신을 감시하고 있어요"라고 말하는 것 |
| 플레이어가 모르는 클리어런스 레벨 이상 정보 제공 | Level 1 플레이어에게 Keter급 SCP 정보 자동 제공 |
| GM Sealed 정보를 다른 플레이어에게 암시 | 배신자를 향한 NPC의 특별한 태도 표시 |
| Shadow Engine 추적 정보 노출 | 재단의 감시 수준이 올라갔음을 직접 언급 |

**단, 다음은 허용된다:**
- 클리어런스 레벨로 접근 가능한 문서를 요청하면 제공
- 탐지 판정 성공으로 비밀 관련 단서 제공 (내용이 아닌 단서)
- 비밀 미션의 존재 사실 ("이번 챕터에 비밀 임무가 있는 사람이 있을 수 있습니다")

## 3.2 클리어런스 위반 처리

플레이어가 자신의 클리어런스 레벨 이상의 정보를 요청하거나 접근하면:

```
클리어런스 위반 처리 흐름:

1. NPC 또는 시스템이 접근 거부 (자동)
   예: "접근 권한이 없습니다. 클리어런스 레벨 3 이상 필요."

2. 위반 시도 자체가 기록됨
   Shadow Engine: 해당 플레이어의 감시 수준 +1

3. 성공적으로 무단 접근한 경우:
   - RP +3 (규정 위반 후 생존)
   - Shadow Engine: 발각 확률 타이머 등록
   - Director Engine: 재단 NPC 대화 태도 변화 암시

4. 발각된 경우:
   - 보안팀 출동
   - 클리어런스 레벨 임시 하락
   - 징계 기록 생성
```

# 4. 배신 메커니즘

## 4.1 배신이란

배신은 **비밀 미션으로 정의된 경우**에만 "공식 배신"으로 처리된다. 다른 플레이어에게 불리한 행동이라도 비밀 미션 없이 행한 경우는 "이해 충돌"로 처리한다.

| 구분 | 정의 | 처리 방식 |
|------|------|----------|
| **공식 배신** | 비밀 미션 "동료 감시·고발"을 달성하는 행동 | RP 변화 발생, 공개 조건 충족 시 파티에 공개 |
| **이해 충돌** | 비밀 미션 없이 다른 플레이어에게 불리한 행동 | RP 변화 없음. 관계 변화만 반영 |
| **자발적 배신** | 어떤 임무도 없이 다른 플레이어를 재단에 고발 | RP -3 (고발자), +5 (고발당한 자) |

## 4.2 배신 실행 규칙

배신 행동을 실행하는 플레이어:
1. GM에게 배신 의사를 비밀리에 통보 (OOC 채널 또는 쪽지)
2. GM은 비밀 미션 조건 충족 여부를 확인
3. 게임 내에서 배신 행동을 선언 (IC 대화, 행동 선언)
4. Director Engine은 행동 결과를 서술 (배신 사실은 즉각 공개하지 않음)
5. 결과적으로 발각될 수 있음 (다른 플레이어의 탐지 판정, 추후 챕터)

## 4.3 발각 처리

배신이 발각되면:

| 발각 방법 | 효과 |
|----------|------|
| 다른 플레이어의 SOC 판정 성공 (DC 15) | 배신 의심 단서 획득. 아직 확정 아님 |
| FDK 판정 성공 + 재단 기록 조회 | 배신 행동의 재단 기록 발견. 배신 확인 |
| 비밀 미션 "동료 감시" 달성 플레이어가 공개 | 즉각 공개 |
| 챕터 종료 시 GM 결정 | GM 서술로 결과에 포함 |

발각 후:
- 해당 플레이어와 발각한 플레이어의 disposition 급락 (향후 협력 판정 -3)
- 발각된 배신자의 다음 비밀 미션은 "감시 강화" 조건이 추가됨

## 4.4 배신의 결과

배신은 단순한 처벌이 아니다. 서사의 한 축이다.

- 배신이 성공해도 배신자는 재단의 도구가 된다 (RP 감소 없이 얻은 RP는 그 대가가 있다)
- 발각된 배신자는 오히려 다른 플레이어들과 결속할 기회를 얻을 수 있다
- 배신당한 플레이어는 분노에서 오는 RP 획득 기회를 얻는다

# 5. Shadow Engine 연계

## 5.1 Shadow Engine이 추적하는 SCP 모듈 전용 항목

```yaml
# Shadow Engine npc_agendas 확장
scp_shadow_agendas:
  - agenda_id: "shd_agd_retirement_watch_01"
    agenda_type: retirement_monitoring
    target_player_id: string
    description: "재단 감시 수준 자동 상승"
    progress: int              # player.retirement_points와 연동
    trigger_thresholds:
      - threshold: 30
        action_type: director_hint
        description: "재단 NPC들이 플레이어를 조금 더 유심히 보기 시작한다"
      - threshold: 60
        action_type: world_effect
        description: "요주의 인물 마킹. NPC 태도 변화 WorldEffect 발행"
      - threshold: 80
        action_type: shadow_scene
        description: "처분 명령 또는 아메스틱 투여 검토 타이머 등록"
```

## 5.2 비밀 미션 진행 추적

Shadow Engine은 각 플레이어의 비밀 미션 진행도를 `npc_agendas`로 추적한다. Mission Engine이 비밀 미션 상태를 관리하지만, **다른 플레이어에게 누설되지 않도록** Shadow Engine이 별도로 격리하여 보관한다.

## 5.3 재단 내부 음모 추적

챕터 진행 중 재단 자체의 숨겨진 계획 (D계급 대량 처분 예정, 실험 프로토콜 강화, 인사 숙청 등)은 Shadow Engine의 `background_events`로 관리된다.

```yaml
background_event:
  event_id: "shd_evt_d_class_purge_01"
  description: "3챕터 후 D계급 전원 처분 명령이 예약되어 있다"
  scheduled_at_epoch: int
  world_effects:
    - effect_type: faction
      target_id: "faction_d_class"
      operation: update
  visibility: gm_only            # 플레이어에게 처음에는 비노출
  cancellation_condition: "D계급 플레이어 중 한 명이 RP 60 달성 전 공식 직원 전환 성공"
```

# 6. GM 전용 정보 관리

## 6.1 GM 전용 정보 목록

GM은 다음 정보를 독점적으로 보유하며 Shadow Engine으로 관리한다:

| 정보 항목 | 설명 |
|----------|------|
| 각 플레이어의 정확한 RP 수치 | 플레이어는 자신의 것만 알고 있음 |
| 모든 비밀 미션 내용 | 해당 플레이어는 자신의 것만 알고 있음 |
| 챕터 내 숨겨진 타이머 | 만료 시 Shadow Event로 Director에 알림 |
| 각 SCP 개체의 진짜 능력 | 플레이어는 관찰과 추리로만 발견 |
| 재단 내부 NPC 음모 | 플레이어는 단서로만 접근 |
| 퇴사 유형 결정 기준 조건 충족 여부 | 플레이어는 결말에서만 확인 |

## 6.2 GM 정보 공개 원칙

GM은 다음 원칙에 따라서만 GM 전용 정보를 공개한다:

1. **트리거 조건 충족:** 비밀 미션 완료, 추리 판정 성공, 발각 이벤트
2. **챕터 결산:** 챕터 종료 시 GM 선택으로 일부 정보 공개
3. **캠페인 피날레:** 모든 RP 이력과 비밀 미션 결과 전면 공개
4. **플레이어 요청 불가:** GM 전용 정보는 플레이어의 OOC 요청으로 받을 수 없음

# 7. 비밀 공개 절차

## 7.1 단계별 공개 흐름

```
[레벨 1: 암시]
Shadow Engine → Director Engine: atmosphere_cue (can_hint)
Director Engine: 서술에 모호한 단서 삽입
(예: NPC가 특정 플레이어를 묘하게 피하는 묘사)

[레벨 2: 단서 획득]
플레이어의 능동적 조사 + 판정 성공
→ 구체적 단서 제공 (내용이 아닌 방향)
(예: "감시 카메라 기록에서 누군가가 당신을 의도적으로 따라다녔음을 알 수 있다")

[레벨 3: 결론 확인]
추리 선택지 시스템 또는 직접 대면 확인
→ 비밀의 내용이 확정됨
(예: 배신자가 누구인지, 재단이 무엇을 숨기고 있었는지)

[레벨 4: 전면 공개]
챕터 결산 또는 피날레
→ GM이 모든 숨겨진 정보 공개
```

## 7.2 비밀 공개 시 처리

NPC Engine이 관리하는 비밀 정보가 공개될 때:
- 해당 NPC의 knowledge.secret → knowledge.public으로 이관
- 관련 플레이어들의 NPC disposition 갱신
- Shadow Engine의 해당 agenda status: revealed 전환

# 8. 재단의 정보 통제 시스템

## 8.1 아메스틱 시스템

아메스틱(기억소거제)은 재단의 최강 정보 통제 도구다. 이 모듈에서 아메스틱은 RP 시스템과 직접 연결된다.

| 아메스틱 유형 | RP 효과 | 게임 효과 |
|-------------|---------|---------|
| **Class-A** | -5 ~ -10 | 최근 2~6시간 기억 소실. 관련 단서 소멸 |
| **Class-B** | -10 ~ -15 | 최근 수일 기억 소실. 비밀 미션 관련 기억 소실 가능 |
| **Class-C** | -20 ~ -25 | 장기 기억 선별 소실. 특정 NPC 관계 기억 소실 |
| **Class-F** | RP 0으로 강제 초기화 | 거의 모든 재단 관련 기억 소실. 사실상 새 캐릭터 |

**아메스틱 투여 트리거:**
- 클리어런스 미달 정보 접근 발각 후
- 격리 위반 목격자 처리
- 퇴사 시도 실패 후 재단 회수

**저항 규칙:** WIL 판정 (DC = 아메스틱 등급 × 3 + 5). 성공 시 투여 효과 절반. 실패 시 전량 적용.

## 8.2 재단 감시 시스템

재단은 모든 직원을 감시한다. Shadow Engine이 추적하는 감시 수준은 게임 내 현실이다.

| 감시 수준 | NPC 행동 변화 | GM 서술 신호 |
|---------|-------------|------------|
| 0 (일반) | 변화 없음 | 없음 |
| 1 (관심) | NPC들이 조금 더 자주 시선을 준다 | atmosphere_cue: subtle_hint |
| 2 (요주의) | 특정 NPC가 플레이어 행동을 기록한다 | atmosphere_cue: direct_foreshadow |
| 3 (고위험) | 미행 확인됨. 이동 제한 가능성 | shadow_event: can_hint |
| 4 (처분 대기) | 처분팀 대기 중 | shadow_event: can_reveal (Director가 서술에 직접 통합) |
